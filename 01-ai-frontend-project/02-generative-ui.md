# AI时代的UI：AI生成，而非人工编写

> Created By [RV](mailto:rodney.vin@gmail.com), and licensed with Creative Commons "[CC BY-NC-ND 4.0](https://creativecommons.org/licenses/by-nc-nd/4.0/)"

**AI时代的UI，是由AI生成，而不是人工编写的。**

这应该是我们澄清的第一个现实，并且是后续所谈一切的前提约束。

脱离了这一约束，各种各样的声音，相互之间，基本就是不在同一频道的对牛弹琴。

如果您坚持认为，你能够以手工编程的匠人精神与AI竞争并获得战略优势，那么请您坚持并继续，我们只有尊重并祝福。

我们基本的认知是：

- 软件行业，不论是前端还是后端，其未来都是AI的。

- 一切应该以AI First为战略目标，以AI Native为架构约束。

### 一. 何谓AI First，AI Native

**AI First，是一种态度。**

这种态度，要求在产品、架构设计时，要把AI放在最重要的位置上去考虑。

AI，是一种被前置考虑的重要能力，而不是事后叠加的模块。

而系统的底层，可能仍是传统的设施、架构。

**AI Native，是一种原生架构约束与能力。**

这种架构，要求AI是整个系统的根基，移去了AI后，则整个架构、系统会完全崩塌。

我们正在构建前端UI体系：

它是AI First的，一切为AI优化，用尽AI的能力是基本前提。

它是AI Native的，AI生成UI，这是基本前提约束也是最终的目标，没有AI整个体系则无法运作。

### 二. 限定AI的能力边界

对于我们所面临的绝大部分场景，AI的能力往往是溢出的。

我们所要解决的问题，不是怎么补上AI的能力缺失，而是约束AI的能力不要泛化到不相干的领域，约束AI的注意力集中在当前特定的问题上，约束AI输出的不确定性。

所以，使用AI生成UI时，所有的一切往往是从如何画个圈，划定一个AI明确不能突破的功能、安全、能力全集起始。

简单来讲，两层含义：

- Metadata层面：定义Catalog Schema，一个结构化规范的目录，明确一种数据结构去定义AI所能使用UI Component、Style Design、Service、DataModel。
- 工程层面：依据上述Schema，创建Catalog，定义当前工程的UI Component、Style Design、Service、DataModel。

具体如何划定AI的能力边界，这是后话，后续再展开。

### 三. 用尽AI的能力

AI时代，不同个体间的竞争优势，往往在于如何利用AI的能力，以及利用的广度与深度。

AI Enabled，使用AI加速已有的工作流程。

AI First，针对AI设计专门的、为UI优化的、自动化的基础设施、工具、流程。

AI Native，以AI生成UI作为基本约束与目标进行架构设计。

这是逐层递进的三种利用AI的方式。

其最终目标，是用尽AI的能力，流程由AI驱动与主导，UI由AI生成而非人工编码，人仅在AI不合适或者不经济的环节介入。

#### 生成式UI，AI主导

UI是由AI来生成的，不论是开发阶段静态生成预置式UI，还是在运行时动态、实时生成UI。

这种角色的转换，实际“生产者”从人类变更为AI，是当前的产业现实。

一个AI-Enabled的团队，即使是基于当前重型的、为人类认知而优化的各种前端框架，去AI Coding，绝大部分的代码也是由AI来编写的。

虽然这种不是AI First、AI Native的重型编程式前端框架为AI带来了巨大的认知负担，导致现有的生成过程基本是一种低效的编程模型：生成->错误检测->修正->再次生成……

在这我们看到一个巨大的gap，需要被填充。

我们需要在满足预置式UI、实时生成式UI这两种场景的前提下，设计一种同一、高效、符合AI认知模型、能力模型的AI Native的前端UI方案。

#### 仅在必要时人工参与

前端UI层面，实际存在大量不合适使用AI去处理的场景。

意图，UI设计所体现的人类意图，只有人类才能提供。如何向AI理清人类的意图，实际是整个AI Native的前端架构下，人类主要的工作。

无论使用手绘或者自动生成的设计稿图形，还是使用自然语言描述，您想做什么，想得到什么，想实现什么价值，这只有系统的设计者才能知道。

划定AI的能力边界，如果由AI自动划定自己的能力边界，这会构成一种罗生门式的衔尾蛇结构，这其中的风险不言自明。

AI的输出，虽然AI可以编写各种自动化测试脚本去验证，但是人类的人工验证是不可或缺的。至少在可见的将来，是不可或缺的。

很多人，有很多的焦虑，觉得AI时代，人类会消失。

人类工程师不会消失，只会自我提升、自我进化，钉在人类不可或缺的环节上，与AI协同工作。

毕竟，自由意志，是人类最大的能力与财富，而AI的本质是一个没有灵魂的机器。

至少，现在如此，捂脸🤦‍♀️。

### 四. 迎合AI的偏好，直面AI的不足

扬其长避其短，这是我们使用一个工具时，基本的态度。

是的，AI仅仅是一个工具。虽然它比绝大部分人类都强大，但是它的本质还是一个工具。

AI有其明确的能力边界与偏好，顺着捋则自然顺畅，逆着来则后患无穷。

#### 提供结构化输入

AI是偏好结构化的文本输入的。

这是一个浅显、不言而喻的通识，没有必要展开去讲。

所以，提供给AI的所有输入，必须是结构化、带有Schema校验的文本。

更明确点，应该是JSON以及JSON Schema。

这个要求，实际是在约束，我们提供给AI的Catalog Schema应该是JSON Schema格式，而Catalog本身，则应是JSON。

#### 非确定性输出必须被强校验

上下文中我们所讲的AI，实质是基于LLM的AI Agent。

基于LLM，则带来了AI输出的天生的、基于概率的不确定性。

AI是万能的、具有近乎全知的世界模型与知识，但是其基于概率模型的输出，导致的输出不确定性必须被处理。

简单的例如，即使prompt中使用最严厉的约束指明期望JSON格式输出，给出正向、反向实例，我们也不能保证其输出一定是合法的JSON格式，偶尔丢上几个闭合符号，司空见惯，见怪不怪。

在AI生成UI，这个上下文语境下，则是AI输出的数据格式必须被Schema定义严格限定，其实际输出必须使用Schema进行强制校验。

错了，要么程序纠正，要么发回AI重来。

#### 声明式输出期望，而非编程式渲染

要求AI输出对UI的期望，来描述UI上应该有哪些组件，应该使用什么布局器去布局，应该使用什么样式主题。

声明式的期望，是一种数据。对于AI来讲，这是一种没有认知负担的easy任务。

要求AI输出代码，进行过程式的、编程式的渲染，这个事情则有些一言难尽。

首先，现有的重型的、工程化的前端框架，是针对人类认知进行设计的。其之所以存在，是契合人类前后端职责分离、专业细化、分工协作的要求。状态、通信、组件、逻辑、数据、渲染、代码，大量的前置知识约定、运行时各部件的精巧配合，所有这些混为一体。

如果AI生成式UI，基于此种模型则预生成实际就是AI Coding，实时生成则事实上不可能。

AI生成式UI，采用编程式模型，是毫无性价比的邪路。

### 五. AI生成式UI的特性

前边讲了很多抽象的废话。

虽然废话，但是不得不讲。讲在明处，道理是道理；作为隐含的上下文不讲，则道理会变成鸡汤。

设计AI Native的前端框架时，有几个自然而来的架构特性，需要讲在明处 。

这几个特性，虽然下意识中每个人都知道，但是明确的、有意识的认知，则不一定人人都有。

#### Server端驱动

**AI生成式UI，一定是Server端驱动的。**

这是一句无比正确的废话，AI在Server端生成UI，浏览器/Mobile APP在客户端渲染UI。

不需要太多解释。

虽然Server-Driven，到底Driven的是什么，这点并没有一致的共识。

有人在Server端渲染完整UI，有人在Server端仅生成驱动数据，仁者见仁智者见智了。

#### 剥离数据与渲染，生成数据而非UI

从AI Native的角度、从AI便利性的角度，剥离AI的职责，将“渲染”剥离为一种AI可用、可调度的基础设施，而AI只负责生成驱动“渲染”的数据，会大幅降低AI生成UI的难度、以及后续调试、维护的成本。

剥离数据与渲染，在当前前端的技术体系下，不考虑各种大而无当的框架，其对应的基本就是Web Component与双向的数据绑定Data Binding。

这是技术细节，后续再讲。

#### 流式输出，增量更新

基于LLM的AI Agent的输出是流式的，这也是一个不言而喻的常识了。

在生成预置式UI的过程中，我们可以等待，等待AI输入完成，保存为文件，后续一次加载。

而在动态的实时生成式UI的场景下，我们必须直面这种流式输出。

要求用户长时等待，然后一次更新，这是一种糟糕的UX设计。

AI流式输出，Server端Diff后，只增量更新UI的变动部分，才是一种UX、成本、性能约束下合理方案。



BTW：当前生成式UI的现状

1. 基本未考虑应用于预置式UI的问题

2. 存在两个流派，各有利弊：

   - UI Component + 数据驱动

   - Generative Raw UI
