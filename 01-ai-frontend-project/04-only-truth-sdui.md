# 唯一真相在服务端：Server-Driven UI，LLM或者预定义服务

> Created By [RV](mailto:rodney.vin@gmail.com), and licensed with Creative Commons "[CC BY-NC-ND 4.0](https://creativecommons.org/licenses/by-nc-nd/4.0/)"

UI 的唯一真相在服务端。

这句话有两层含义：

1. **运行时**：UI 的布局、内容、交互行为，最终由服务端决定
2. **方法论**：预置式 UI 来自 AI 预生成，动态 UI 来自 AI 即时生成——源头都在服务端

这不是巧合，是同一原则的两个侧面。

---

### SDUI 的前提：谁写出那个 JSON

Server-Driven UI 并不是新概念。2018-2024 年间，Airbnb、Lyft、DoorDash、Uber 都做过投入。但一个尴尬的事实是：**SDUI 从未成为行业主流**。

原因可以列出很多：要维护渲染框架、要管理 Schema 版本、要跨团队对齐组件契约。但这些都不是最根本的。

最根本的问题是：**"谁写出那个描述 UI 的 JSON"** 这个日常问题一直没有好的答案。

在 AI 出现之前，写 UI 编排的人只有两个选择：

- **后端工程师写** — 不情愿，这不是他们的核心技能
- **前端工程师转 BFF 写** — 人没少，只是换了位置写代码

两种方案的共同困境：每一屏 UI 的编排都是**人力成本**，而且随着业务增长**线性增长**。

所以 SDUI 在没有 AI 的时候，经济账是这样的：

```
收益（绕过发版 + 动态实验 + 跨平台一致）
vs
成本（框架维护 + Schema 管理 + 每屏 UI 编排的人力）
```

对大多数团队，这账算不过来。

---

### AI 改变了成本结构

AI 的到来，改变了这个等式中最关键的一项：

```
收益（绕过发版 + 动态实验 + 跨平台一致）
vs
成本（框架维护 + Schema 管理 + AI 生成蓝图的算力）
```

**"每屏 UI 编排的人力"变成了"AI 生成蓝图的算力"**，从线性增长变成了边际递减。

这不是渐进式的改善，而是结构性的变化：

| 成本项 | 无 AI 时代 | AI 时代 |
|--------|-----------|---------|
| 渲染框架维护 | 需要 | 需要（不变） |
| Schema 管理 | 需要 | 需要（不变） |
| **UI 编排** | 工程师逐个手写，每屏成本 × n | AI 生成，算力成本随规模摊薄 |
| 调试排查 | 前后端联调，定位哪一端错了 | 蓝图可审查、可回放、可验证 |

当 UI 编排从人力密集型变为算力密集型，SDUI 的经济模型就彻底翻转了。

---

### 历史的注脚：Spotify HubFramework

Spotify 的 HubFramework（2019 年弃用）常被引为"SDUI 不行的证据"。但实际上它验证的是另一件事。

HubFramework 的问题不是 SDUI 不行，而是它做了一个**只有 iOS 的 SDUI**：

| 方面 | 正常 SDUI 的收益 | HubFramework 实际做到的 |
|------|----------------|----------------------|
| 跨平台统一 | 一套逻辑控制 iOS + Android | ❌ 只能管 iOS |
| 绕过发版 | iOS + Android 同时受益 | ❌ Android 还得各自搞 |
| 框架维护成本 | 摊到两个平台 | ❌ 只摊到一个平台 |

跨平台是 SDUI 核心价值之一，HubFramework 主动放弃了它。同期 Spotify Web Player 重写选择了 React + Redux，且明确"避免自建框架"——对 Web 端来说，刷新即更新，SDUI 没有额外价值。

HubFramework 的教训不是"SDUI 不行"，而是：

1. **只做一个平台的 SDUI，等于主动砍掉一半收益**
2. **没有 AI 的情况下，UI 编排的人力和框架维护成本，需要足够大的规模才能摊平**

后者正是 AI 所改变的东西。

---

### SDUI + AI：为什么是现在

在 AI 能够生成结构化 JSON 的今天，之前卡住 SDUI 大规模采用的问题逐一松动：

| 过去的卡点 | 现在的情况 |
|-----------|-----------|
| 每一屏 UI 编排都需要人写 | AI 生成，人类审核 |
| 新组件类型需要等客户端发版 | AI 可基于 Catalog 原语组合表达，注册新组件类型的频率降低 |
| Schema 版本兼容麻烦 | AI 可理解版本边界，自动适配降级 |
| 调试困难 | 蓝图由 AI 生成，可审查、可追溯、可回放 |

这回答了"为什么 SDUI 在 2025 年之前没有流行"的问题：**不是 SDUI 这个想法有问题，而是它的前置条件在 2018-2024 年不成立——那时候没有能低成本生成结构化 UI 描述的东西。**

---

### Kree4X 的位置

Kree4X 不是给"没有 AI 的 SDUI"设计的。它的架构决策——Service-Call-Driven（无消息层）、Catalog/Blueprint/Interaction 分离、WebComponent 原生渲染——所有这些都是在回答同一个问题：

> **如果 AI 是 UI 编排者，架构应该长什么样？**

| 决策 | 为什么为 AI 优化 |
|------|----------------|
| 无消息层，Service-Call-Driven | AI 生成的是"调用什么 service"而不是"发什么消息"，语义更清晰，生成空间更可控 |
| Blueprint 与 DataModel 分离 | AI 可以独立更新 UI 结构或数据，不需要理解双方耦合的复杂状态 |
| Catalog = 白名单机制 | AI 的输出空间被 Schema 约束，不会生成 Catalog 里没有的组件 |
| WebComponent 原生渲染 | 不需要 Virtual DOM 等重型框架，渲染器足够薄，AI 生成的蓝图直接映射到原生元素 |

SDUI 的老问题——"谁写那个 JSON"——在 AI 时代有了答案。Kree4X 的假设是：**既然 AI 来写，那架构就按 AI 的习惯来设计。**

---

### 小结

- SDUI 没在 2018-2024 大规模流行，不是因为架构有问题，而是**人力编排 UI 的成本超过了收益**
- AI 将"UI 编排"从人力密集型变为算力密集型，改变了 SDUI 的经济模型
- Spotify HubFramework 的教训是"单平台 SDUI 不行"，不是"SDUI 不行"
- Kree4X 是从第一天就为 AI 编排设计的 SDUI 架构
